# The Knowledge Game: Deployment Specification v2

Merges the original deployment guide with the refinement pass. Where they disagree, this document governs. Where this document and the upstream protocol disagree, the protocol governs; this spec is a synthesis, one viewpoint's reach, and the build grounds itself in the protocol, not in this document's authority.

Upstream: A-Viable-Fork/epistack. The upstream repository is never modified by this project. Every upstream capability this spec relies on is **built and oracle-backed on upstream main** as of the pin: snapshot emit (`build/emit-snapshot.mjs`), on-device kernel and gate (`api/client-api.mjs` over `api/providers/local-provider.mjs`), read with derived grounding, propose with receipts, contribution bundles with content-derived identity (`api/contribution.js`), type fork and contest (`api/fork.js`, `api/contest.js`), parameterized recompute (the provider takes `{state, sources, kinds}`), the scaffolder (`scaffolder/new-kernel.mjs`), and the ecosystem contracts (`docs/ecosystem-guide.md`, `docs/clients.md`). What is deliberately not relied on because upstream marks it specified: the credential seam [4.2], the patch ledger [4.5], the standing economy, live remote providers.

## 1. What this is

A mobile-first, installable PWA: a community feed reader and community founder for the EpiStack protocol. Pick a community, see its graded claims, scroll, interact, contribute, switch communities, found one. The minimal surface that makes the protocol lovable, built as a protocol demonstrating itself: the repository follows the six-stage kernel workflow while being built, carries its own governance kernel, and its first community's claims are about EpiStack itself, for the competition.

Two orthogonal tracks run through the plan and must not be conflated:

- **The workflow track** governs the repository's own accounting: Stage 0 framing, Stage 1 generated governance kernel, Stage 2 bare claims, Stage 3 grounding checks, Stage 4 embedded references, Stage 5 invitation. Each stage is a real committed state; the history is part of the demonstration.
- **The product track** governs what ships: Phase A (the base reader), Phase B (community founding in-app), Phase C (the first community, dogfooded). Phases map into Stage 3 of the workflow track as they build and ground.

## 2. Architecture invariants (Tier 0, unchanged from v1)

One trust boundary, one direction: periphery -> API/providers -> kernel. The periphery never imports the kernel or store directly; the API is the sole membrane; the vendored kernel stays pure and headless. One canonical schema source: the pinned upstream; no parallel claim schema is ever invented locally. Public kernel data and private personal data are structurally separate; personal data never enters a public patch. No hidden ranking objective; ranking never changes standing, grades, receipts, robustness, or support structure, and a mechanical check proves it. No undeclared network egress; no third-party analytics; all release capabilities declared in machine-readable manifests. Gate passage, admission, and semantic acceptance are three states rendered distinctly; no state ever implies the next. A passed gate receipt is never labeled true or validated. All inherited substrate code is pinned, hashed, and attributable; local departures are explicit patches with named checks. Gaps are ledgered, never hidden. No tokens, no cryptocurrency, no financial incentives, no personal data on any public ledger. This client is the first third-party client, never a privileged one: it holds no capability any client with the snapshot lacks, claimed as a governance claim and grounded two ways: the import-graph oracle (the app reaches nothing past the public membrane) and the second-client conformance check (an independent minimal client, built in CI from the vendored public api alone, reproduces every app capability against the same published artifacts). Communities, snapshots, cards, and contribution targets produced by this app are client-neutral artifacts containing no reference to this app; admission judges bundles, never the client that produced them.

## 3. Upstream dependency strategy (unchanged in substance)

Pin the exact upstream commit; **provisional pin now, re-pin at the July 19 freeze** so the lock file names the submitted state. Machine-readable lock file: upstream identity, commit hash, acquisition date, imported files with hashes, local patches (expected: none), license, protocol identity. Strategy order: pinned submodule plus deterministic extraction; pinned subtree; vendored minimal substrate with reproducible import script and integrity manifest. Never copy upstream code to multiple locations; never edit vendored files; a CI check verifies vendored hashes against the lock on every build. The repository can always answer: which protocol version, which code inherited, which changed, which changes are presentation-only, which alter local policy, which would break composition.

## 4. Repository layers

`upstream/` pin and extraction records. `vendor/` imported substrate. `kernel/governance/` the app's own governance kernel (scaffolder-generated, never hand-invented). `api/` the membrane: provider adapters, the credential seam, the contribution path. `periphery/` the PWA and every fallible view and producer. `vault/` the minimal personal-data interface. `trellis/` governing-trellis.md, design-axioms.md, parameters-register.md, status-ledger.md, sorry-ledger.md, exclusion-reservoir.md. `manifests/` capability, network, build provenance. `build/` checks and generated artifacts. `docs/`. `app/` built PWA output.

## 5. The hot-swap seams (new in v2, named as first-class design)

Four upstream capabilities are specified-not-built. The app does not wait for them and does not fake them; it builds against explicit interfaces whose v1 implementations are honest nulls, so that when upstream lands each capability, the upgrade is a swap behind a seam, not a restructure. Each seam is a governance claim ("this boundary exists and nothing crosses it except through the interface") grounded by an import/contract check.

- **Credential seam.** Interface shaped to upstream [4.2]: who may perform which action at which identity grade, time-locked, reading requiring none. The seam's subject is the **presented identity**, never the identity: a presentation is a predicate over the agent's attestation graph (this key; a member of group G; a member of this community; the empty predicate), the presented grade is the grade of the predicate's own claim, and thresholds evaluate the presentation, so the full menu from verified individual down to floor anonymity is an agent's right by interface shape from day one. The binding point for presentation proofs on acts is the contributor field on records, an honest unverified string until the seam wakes, at which point it becomes a presented-identity reference verifiable against a key or a member-set commitment. v1 implementation: open (any action the community's parameters allow without identity), with GitHub identity riding the PR path for contributions, which also means v1 offers no anonymity at the transport layer: the presentation machinery, once built, guarantees the graph contains no linkage, and the GitHub-native admission path still names the pusher, so anonymity claims are scoped to the graph, agent-side de-anonymization is always unilateral (an agent can later prove which acts were theirs; absent an opening authority no one can prove it about them), and transport-level anonymity (relays, submission intermediaries) is its own ledgered gap. When upstream credential.js and identity-as-graded-claim land, the null swaps for the real evaluator and community identity thresholds simply begin evaluating. The founding flow (Phase B) already **collects and stores identity-threshold parameters** in the community's parameter record, marked inactive-until-credential-seam-active, so no community needs re-founding when standing arrives. This is the hot-swap the spec is designed around.
- **Patch history seam.** v1: content-derived bundle identity per contribution (built upstream). When the patch ledger [4.5] lands, bundles gain durable history without changing identity.
- **Provider seam.** v1: static snapshots through the local provider, gate on device. When live remote providers land, they arrive behind the same client contract; the periphery cannot tell and must not be able to tell.
- **Standing economy seam.** Absent entirely in v1. No stake, no slashing, no incentives. The parameter records reserve the field; nothing reads it.

## 6. Phase A: the base

A feed reader over one snapshot. Fetch `<id>.snapshot.json`, verify its hash, load through `createLocalProvider`, render a vertical feed of typed claims with grades computed on device. Development fixture: the upstream corpora emitted as snapshots, so the base has real content before Phase C exists.

Card disclosure, trimmed to three levels: (1) statement, type, standing, origin, why-selected; (2) supports, challenges, provenance, open gaps; (3) actions: propose support, propose undercut, propose qualification, contest, fork the type, export contribution. The elaborate Level 3 of v1 (crossing history, alternative frames, crux candidates) is specified, deferred.

Ranking is a personal view-side function. The objective vector is always visible; every card answers "why am I seeing this"; behavioral observation of scrolling is **toggleable and default off**, opt-in as typed local observations, never silent. The null objective is defined: with ranking off, the feed is deterministic, grounding-sorted then recency, declared as a Tier 1 default in the trellis. The objective list from v1 stands (learn efficiently, unresolved cruxes, weak-but-influential, strongest opposition, neglected, recent changes, breadth, novelty, validation-expertise match, followed topics, engagement sufficient to continue reading), engagement present only as a declared, chosen component. The epistemic-cost report ships in the base because upstream already supports it: the same records recomputed under another community's tables, "your objective surfaced 12 claims; 8 are contested under Community B's parameters." Structural reflexivity, not paternalism.

No likes. No popularity-derived standing. Gate/admission/acceptance rendered as the three-state ladder on every contribution-bearing surface.

**The extension seam**, the app absorbing other people's tools as first-class citizens of the view market. An extension is a content-addressed module (install is pinning a hash) in one of three shapes: an objective component or ranker, a renderer lens, or a composed workflow mode. The contract is the theorem the app already obeys: rankers are pure functions from frozen records and a vector to an ordering, and install-time conformance runs the ranking-separation fuzz against the candidate, so anything loaded provably cannot move standing, and anything that could does not load. Extensions execute sandboxed (worker, no DOM, structured data in and out) under their own capability manifests, and network access is capability-scoped: an extension may declare exact network destinations in its manifest, shown at install and user-approved, enforced per-extension by the egress machinery; extensions declaring none run fully offline as before. This is the deliberate, visible seam change the earlier sterility language reserved: with it, the wallet exclusion is a registry policy of this deployment rather than a construction (this app's registry carries no financial extensions), stated here so the downgrade is chosen and named rather than drifted into. **Assistant extensions** are the change's purpose: an assistant is a prompt pack plus local graph slices in through the structured channel plus one declared inference destination with the user's own key, a market of assistants rather than a built-in one, each an unprivileged producer whose output enters nothing except through draft, gate, and a human's hand. Where a community runs a treasury, its books are typed records and the contestable dashboard renders them like any quantitative claims; this app shows market-layer facts as claims and never holds market machinery. Renderer extensions enhance and must degrade to the standard rendering, so the closed presentation set remains the floor and no extension can make a kind invisible. Discovery is an extension registry that is itself a kernel whose claims describe extensions, the directory-as-kernel convention reused. The named first-wave extension examples: teaching rankers, and the **contestable dashboard**, a renderer lens over quantitative kinds turning any metrics-bearing kernel into a dashboard where every displayed number is a claim, standing shown, provenance one tap down, contest and fork as live actions; the institutional counterpart to the consumer feed. The app's own learn-efficiently ranker ships as the first extension, loaded through the public seam, the unprivileged discipline applied inward. Learn-efficiently is the default objective, and onboarding's first act is asking what the user wants to learn.

**Gate feedback is collaborative by law.** A refusal never renders as rejection alone: the draft screen renders the receipt's content as the structure present, the structure missing, and the support shapes that would ground the proposal, which the gate already computes. Punitive gates breed gaming; gates that show the path breed grounding.

**Standing-motion alerts**, specified: watch any claim, be signaled when its computed grade moves or a watched high-blast-radius dependency collapses. Alerts report grade motion and only grade motion, are computed locally from snapshot diffs, and are unsuppressible in the sense that the motion is a fact of the graph; the notification is the honest retention mechanic.

**The type filter**, universal across every claim-list surface: a set-selector over the kinds present in the current graph, with `untyped` as a first-class pseudo-type so crossing arrivals are filterable rather than invisible. The filter composes with ranking (filter selects the set, the objective orders it), persists per community in the vault, and obeys the visibility discipline that governs the objective vector: **a filter states its exclusions**, always showing what it is hiding by kind and count, so filtering never becomes silent occlusion. When the upstream comment kind arrives with the freeze re-pin, comments render as threads at Level 2, filtered like any kind (comments-off is claims-only reading), with a promote-to-claim action lifting a comment into the draft path; comments are grounding-inert upstream by construction, so none of this touches the separation claims.

PWA shell: installable manifest, service worker, offline shell, deep links as claim hashes (client-portable by construction), reduced motion, screen-reader labels, visible sync state, visible community origin, no analytics. Capacitor: only after the vertical slice works, only if the dependency boundary survives, and not before freeze.

Checks landed with Phase A: import-graph oracle (periphery imports only through api; kernel imports nothing from api, periphery, vault); ranking/standing separation as a property test, the no-grade-motion theorem's third instance (perturb the objective vector arbitrarily, every grade and certificate byte-identical); objective visibility and reproducibility; no undeclared network request succeeds under test; snapshot hash verification on load; offline shell works; deep links resolve; basic accessibility; and the read half of the second-client conformance check: a headless script using only the vendored public api fetches the same snapshot, verifies the same hash, and derives byte-identical grounding for every claim the app renders.

## 7. Phase B: founding a community in-app

The act the ecosystem guide assigns to a community founder, surfaced as a flow. Steps, each producing a real artifact:

1. **Frame.** Name the domain, the community's purpose, what its kernel will and will not govern. Becomes the community's Stage 0 record.
2. **Type.** Adopt shared type bundles by pinning hashes; author local kinds where honest ones are missing. The flow never silently mislabels: where no honest kind exists, it stops and surfaces the type decision, exactly the upstream discipline.
3. **Set the free parameters.** The flow surfaces **exactly the parameters upstream's parameters-register marks free, and only those**, one canonical source: typing policy details, admission policy, identity thresholds per action (stored now, inactive until the credential seam activates, per Section 5), ranking defaults offered to members, the null-objective order. Composition-required parameters are displayed, fixed, and not editable, with the register cited as the reason.
4. **Generate.** The kernel is produced through the scaffolder contract (vendored `new-kernel.mjs` and its config schema), never a hand-built parallel. Its identity, adopted kinds, sources, and Stage 0 axioms are explicit; its own check is generated with it.
5. **Publish.** Emit the snapshot with its hash, generate the community card, and produce the GitHub-native scaffolding: a repository layout, Actions running schema and gate checks on incoming PRs, Pages serving the snapshot. Zero-cost, no server, no database. The card carries: the **governance-hash**, the one named hash over the community's canonical parameter record (pins, identity thresholds, standing-economy fields, null-objective order, and the **corpus content license**, a first-class community parameter declaring the terms its claims carry: gift, attribution-required, share-alike, non-commercial, or a named commercial term, so whether a fork owes anything is determinate by reading the source's declared license, enforced legally and normatively rather than mechanically, and traveling visibly with every fork), which is the community's durable content-addressed identity, with `kernel_id` demoted to a human label over it exactly as kind names are labels over type-hashes; the snapshot hash; fetch locations as transport hints; pinned type hashes; the **member-set commitment**, nullable until a community publishes one, the verification anchor for level 2 and 3 identity presentations; the contribution target; and the protocol identity. The card format follows the ecosystem guide; the card emit is implemented here app-side until upstream lands one, and marked as a local implementation of an upstream-specified format.

Contribution path, unchanged from v1 in shape and now built on upstream primitives end to end: draft -> schema validation -> local gate decision -> contribution bundle with content-derived identity -> export -> PR against the community's contribution target -> target's Actions re-run the gate -> admission is a merge, semantic acceptance is people. A pasted citation never becomes an independent confirmation; independence is attested and attributable. When a draft derives from a published source, the draft path captures typed attribution targets (an external registry id, a work-and-author record, a name-plus-work-hash pair) alongside the forced provenance bond, so transcribed work is accrual-ready for any community's market layer without the app holding any market machinery itself. Nothing auto-merges on gate passage.

Phase B lands the write half of the second-client conformance check and the neutrality check. The independent minimal client proposes through the gate, exports a contribution bundle, and that bundle passes the same Actions admission path as one the app produced: admission provably judges bundles, not clients. The neutrality check greps every artifact the founding flow emits (kernel, snapshot, card, contribution-target scaffolding) and fails on any reference to this app: a community founded here belongs to no client, including this one.

## 8. Phase C: the first community

Found, through the Phase B flow, **the EpiStack competition community**: the community whose domain is the protocol and this submission, whose members are the people already in the room evaluating it, and whose claims are about EpiStack itself. Its initial claims are the self-grounding set: claims about the app's own structure and governance (the Section 2 invariants as submitted claims), and claims about the protocol the competition is judging. Entered bare, typed honestly, graded at their honest floor by the gate, awaiting the verification that only the community performs. The kernel describes itself from day one; the community's verification is the product.

The snapshot publishes at a stable URL; the shipped app points at it by default. Discovery beyond the default is a community-card list, app-side, exportable and importable (the OPML move); the directory-as-kernel convention is specified for later, not built here.

**Deadline hedge, explicit:** Phase C is app-independent in an emergency. The community kernel can be authored with upstream tooling directly (scaffolder, gate, emit-snapshot) and published on Pages without a single line of the app existing. By July 19 the submission therefore carries, in guaranteed descending order of ambition: the app pointing at the first community; the first community's published kernel alone as the requirements-as-claims exhibit; the claims themselves as a corpus. The prefix is shared; work on any tier advances all three.

## 9. Governance kernel, trellis, and embedding (compressed from v1, obligations unchanged)

Stage 0 axioms as first-class claims, committed before anything else. Stage 1: the governance kernel generated, not invented; where software evidence (test execution, reproducible build, static analysis, observed network behavior, capability manifest, artifact identity) lacks an honest upstream source class, author **local source classes** in the governance kernel, no substrate fork, promotion to shared substrate deferred until a second deployment wants them, per the retail-to-wholesale doctrine. Stage 2: the bare governance claims (ranking cannot modify standing; the active objective is visible; profile data cannot enter public patches; no undeclared telemetry; the periphery imports no kernel module; the client is unprivileged; declared artifact hash matches built artifact; providers swap without presentation changes; the four seams of Section 5 hold), committed at honest floor, never inflated. Stage 3: Phases A through C build and ground them; the gate computes every standing; insufficient evidence leaves a characterized gap with a closing condition. Stage 4: code and manifests reference the claim ids they are governed by; a check proves every reference resolves; a blast-radius report maps claims to affected code and tests. Stage 5: the invitation, leaving constitutive community choices open, claiming no community that does not exist.

Trellis files as in Section 4, with Tier 0 as Section 2, Tier 1 defaults (null-objective order, disclosure defaults, opt-in observation), free parameters, and the ledgers. No constraint is called checked until a named check enforces it.

Manifests: source commit, upstream pin, build command, artifact hash, protocol identity, provider contract, allowed network destinations, telemetry policy (none), personal-data storage, ranking execution location (device), permissions, profile upload policy (never), contribution export behavior. Hashes prove artifact identity, not ethical behavior; documented as such. Compliance = artifact matches hash, provenance present, manifest valid, and the governing claims hold the required standing under local policy.

## 10. Deferred, explicitly

**Media, the named frontier.** Photos and video enter as typed records referencing content-addressed blobs by hash; the graph carries references, never bytes, so kernels, snapshots, and the crossing are untouched by media size, and blob hosting is a community-declared, community-priced concern (images near-free at static-host scale, video honestly not). Evidentially, a bare media object sits at the floor: what lifts it is its provenance attestation graph, signed capture chains and content credentials (C2PA maps nearly one-to-one onto attestation claims about a hash), graded and contestable like all attestation, so authenticity is computed rather than moderated, producer-agnosticism extended to cameras. Takedown degrades gracefully by construction: hosts remove blobs, references render unavailable-by-hash, the graph never carried the content. Media kinds are an upstream schema addition after the freeze; provenance attestation is specified against existing machinery; nothing current depends on any of it.

Specified and not built in v1, each with its seam or its ledger entry: the full vault (v1 keeps the essential: local-only, explicit interfaces, export and delete, provenance log; encryption where practical); the manipulation audit and inference governance; the transparent-model inference records beyond declared and observed basics; multi-community federation UI and crossing-history views; live remote providers; Capacitor; the directory kernel; workflow modes beyond the base objectives (evidence, crux, adversarial, recent, validation modes are specified as views and arrive after the base); credential enforcement; the standing economy.

## 11. Final deliverable

As v1's list, plus: the four seams demonstrated (each null implementation swappable under its contract test), the second client shown reproducing the app's full capability set against the published artifacts, the founding flow's parameter surface shown equal to the register's free list, the first community's card and published snapshot, and the three-tier deadline hedge status. The status report divides built-and-checked, built-but-manually-governed, bounded follow-up, plural community-policy choices, open research. Do not claim a community exists beyond the people it demonstrably has; do not claim semantic validation where only gate passage exists; do not claim a hash proves safe behavior.

This repository standardizes what must be shared for its artifacts to compose. It does not standardize what should remain local: personal objectives, semantic judgment, participant governance, validation incentives, identity thresholds, or community purpose. Founding a community means setting those, and the app's founding flow is that sentence made into a screen.

## 12. The registry convention (Phase KG-12 amendment)

Section 10 deferred "the directory kernel"; this amendment builds its artifact-facing half. A **registry** is a community kernel whose domain is artifacts (extensions, skins, components, clients, and communities themselves), founded through the identical founding flow every community uses, never a bespoke mechanism. Its claims are typed by two adopted kinds:

**contract-bundle** (ceiling constitutive, grounded by adoption exactly as a vocabulary declaration is): a claim whose extensions carry `{interface_identity, required_oracle, ceiling_statement}`. `interface_identity` names a specification document by its content hash; `required_oracle` names, by artifact hash, the executable (or CI job) that tests conformance to that interface; `ceiling_statement` states in plain language what passing that oracle actually warrants, and never more. Authoring a contract composes these three references; it never embeds executable code in the claim itself.

**artifact-card** (ceiling checked, grounded by real conformance evidence exactly as a mechanical measurement claim is): a claim whose extensions carry `{artifact_hash, artifact_kind}` (kind one of extension, skin, component, client, community) and, when a contract applies, `contract_hash`. Its `checking_records` cite `(artifact_hash, contract_hash, oracle_hash, result)`, real evidence never asserted without that citation.

**Conformance is recomputable or it is not evidence.** For an oracle that executes in the sandbox (the ranking-separation fuzz, a skin's own contract assertions), the app offers a live re-run reproducing the cited result; for an oracle that only runs in CI, the citation renders honestly as "recomputable in the repository," never as app-verified. **Ceilings hold**: an artifact-card's grade warrants conformance under its cited oracle, nothing broader; no registry surface uses the words safe, good, or trusted about an artifact. **The app lists its rivals**: a registry's client-kind entries include this app's own, graded by the identical conformance pair, with no privileged placement.

**The mirror** extends the same convention outward: the protocol's own repositories (epistack's decomposed documents and vocabulary kernel, this repository's own governance kernel) join the app as ordinary browsable communities, fetched as snapshots by hash like any other, their contribution paths reachable through the same draft-and-bundle gesture as everything else. A promotion proposal read there (adopt X to shared) is an ordinary claim, argued in the open and decided by which communities choose to pin it, never a privileged act this app performs on the protocol's behalf.

## 13. The optional account layer (Phase KG-14 amendment)

Section 5 named the credential seam's full menu, from verified individual down to floor anonymity, as an agent's right by interface shape from day one, with v1 offering the null evaluator (open, no identity required) and the presented-identity binding point (the contributor field) waiting on upstream's real evaluator and member-set commitments. This amendment builds everything that menu already allows without waiting on either.

**An account is a local keypair, never a login.** The app generates it (WebCrypto, extractable so the reader holds their own key, exportable and irrevocably deletable), and the account id is a content hash of the public key alone, a name rather than a claim of trust. No server issues it, no server verifies it, nothing syncs it. Reading requires none of it: no read-path module (the community loader, the feed order, the ranking or filter logic, the card renderer) references account state at all, mechanically enforced by `build/check-accounts.mjs`'s own static scan.

**Signatures are optional and additive.** A detached signature over a contribution bundle's own content-derived `contribution_id` travels as a sibling artifact, never mutating the bundle it signs; an unsigned bundle is byte-identical and fully valid whether or not a signature ever exists for it. Anyone can verify a signature today; no gate requires one yet, since verification and admission remain distinct acts exactly as Section 7's contribution path already holds them apart.

**The presentation menu renders its true current state.** Of Section 5's four levels (this key; a member of a group; a member of this community; the empty predicate), the first and the last are functional now: "this key" signs with the account's own key, and "floor" generates and discards a one-time keypair needing no account at all, genuinely unlinkable to it. The two membership levels render, visibly, as disabled, with the plain reason: membership presentation awaits the credential seam's real evaluator and a published member-set commitment, neither of which exists upstream yet. Nothing here is greyed out pretending to work, and nothing functional is hidden behind a wall.

**Self-custody is this deployment's stated choice, not an ecosystem law.** The credential seam's interface shape permits a custodial or provider-synced client just as readily as this one; this deployment holds keys locally by design, states so plainly at the point a reader would otherwise wonder, and the extension sandbox's own capability scoping keeps any installed extension from doing otherwise on this deployment's behalf.

## 14. The three rooms as communities (Phase KG-13 amendment)

The mirror convention (Section 12) extends to the competition's own subject matter. The LHC (black holes), eggs (nutrition), and covid (pandemic origins) corpora already exist as grounded kernels in epistack; this amendment emits each as a snapshot and founds each as an ordinary browsable community in the switcher, exactly as any mirrored repository is. **The rooms are epistack's communities, not this app's.** Each room's card names its own contribution target inside the epistack repository, at the pinned commit its snapshot was emitted from; a contest or correction on a room claim is a pull request there, re-run through epistack's own gate, never a write this app accepts on epistack's behalf. Client-neutrality holds unchanged: this app hosts no room content of its own, fetching each room's snapshot by hash like any other mirrored artifact.

**Grades are the rooms', recomputed, not asserted.** Each room's snapshot is emitted by dynamically importing epistack's own unmodified builder for that corpus and staging its output the same way any kernel snapshot is staged; `build/check-rooms.mjs` re-runs that same builder fresh and compares a sampled claim's earned grade against the same claim recomputed from the written snapshot, so a passing check is evidence the persisted snapshot reproduces what epistack's own computation produces, not a value this repository merely asserts.

**Not every corpus is a single store, and the honest unit is emitted rather than forced.** The LHC and covid corpora each build through one builder returning one claim store, staged directly. The eggs corpus does not: it builds three separate domain stores (nutrition, environment, economics) plus a cross-domain composite layer that grades a claim by citation across store boundaries, a shape this deployment's snapshot format has no representation for. The eggs room emits the union of the three domains' own already-graded claims and links, re-decided together from one genesis over their one shared source and kind table; the composite layer itself is not part of this snapshot, stated as such in the room's own emission rather than force-fit or silently dropped.

**The competition's case-claims walk into their rooms.** Where the competition community's own decomposition names a corpus as the subject of a claim, a walk button carries a reader from that claim into the corresponding room's entry point. None of the competition's case-claims cite a specific claim inside a room, only the room's own checker script; the walk accordingly always lands on the room's entry point, never a fabricated claim-level anchor the room does not carry. `api/room-walks.js` is the one list both the card renderer and `build/check-rooms.mjs` read, so a claimed walk target can never silently drift from a room that actually exists.

A governing claim these mechanics warrant is one sentence: reading requires no account; accounts are local keypairs; signatures are optional. It is stated here rather than entered as a new numbered governance claim this phase, since grounding it as claim-21 would mean re-opening Stage 2's own closed claim set outside the disciplined front-door process Phase A through KG-4 built for exactly that purpose; `build/check-accounts.mjs` already grounds the sentence mechanically, and a future phase remains free to enter it formally if a reason to do so arises.
